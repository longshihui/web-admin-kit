# API 设计规范

本文件用于约束函数、组件、hooks、模块导出和公共能力的 API 设计。具体包入口和发布策略以项目级规范为准。

## 设计原则

- API 应优先表达业务语义，而不是暴露内部实现细节。
- API 名称、参数、返回值和错误行为一旦对外提供，应保持稳定。
- 新增 API 前先确认是否能复用或扩展已有能力。
- 不为单一特殊场景设计过度宽泛的 API。
- 不用布尔参数堆叠复杂模式；存在多个配置项时优先使用具名参数对象。

## 命名与语义

- 名称应直接表达能力、动作或状态，不使用空泛命名。
- 成对操作命名应保持一致，例如 `open` / `close`、`start` / `stop`、`enable` / `disable`。
- 返回布尔值的函数优先使用 `is`、`has`、`can`、`should` 等前缀表达语义。
- 异步 API 的名称或文档应能体现异步行为，避免调用方误判执行时序。

## 参数设计

- 参数数量较多、存在可选项或未来可能扩展时，优先使用参数对象。
- 可选参数应有明确默认值和默认行为。
- 不要让调用方传入内部状态结构；应暴露面向使用场景的参数。
- 参数校验失败时，应有明确行为：抛错、返回失败结果或忽略，并保持一致。

## 返回值设计

- 返回值应稳定，避免根据不同分支返回结构差异过大的结果。
- 需要表达失败原因时，优先使用可辨识的错误类型、错误码或结构化结果。
- hooks 或组合式 API 返回对象应职责单一，不返回过大的杂项对象。
- 组件事件 payload 应语义明确，避免直接暴露内部临时状态。

## 兼容性

- 删除导出、重命名字段、修改类型签名、修改默认值都属于兼容性风险。
- 修改错误抛出、异步时序、边界输入处理也属于用户可感知行为变化。
- 必须变更已有 API 时，应优先考虑保留兼容层、迁移路径或清晰说明。
- 不要通过静默改变行为来修复设计问题；应明确记录变化和影响范围。

# 业务开发代码规范

## 表单

- 表单页优先拆成页面编排层、表单模型和校验 hooks、子业务区块组件。
- 表单数据应内聚在表单组件内部，父组件只负责打开、关闭、提交结果编排等页面级职责。
- 表单校验规则应靠近表单模型或专属 hook，不要散落在模板附近。
- 复杂字段映射、提交前数据装配、回显转换应拆到 hook 或 `utils/`。

## 表格

- 表格页优先拆出查询条件状态、表格列定义、列表加载与分页逻辑、页面级跳转和操作编排。
- 表格列定义过长时，不要长期堆在路由页面中。
- 行操作涉及权限、状态机或二次确认时，应抽成明确的业务操作编排。

## 弹窗与抽屉

- 多个弹窗或抽屉职责独立时，应拆成独立组件。
- 不要在单页面中用大量互不相关的布尔值统一控制所有弹窗、抽屉、表单状态。
- 弹窗或抽屉内部包含独立加载、提交、校验、回显逻辑时，优先下沉为组件和 hook。

## 接口调用与状态

- 接口调用应明确加载态、成功态、错误态和空态，不要只依赖数据是否为空隐式表达状态。
- 重复请求、快速切换查询条件或连续翻页时，应考虑竞态和旧请求覆盖新结果的问题。
- 页面初始化、刷新、重置、分页加载等动作应有明确入口，不要把请求逻辑散落在多个事件处理函数中。
- 错误处理应靠近请求编排逻辑，避免在模板中堆叠复杂错误分支。

## 权限与条件渲染

- 简单权限判断可以直接用于模板，复杂权限判断应下沉为具名函数、hook 或业务操作编排。
- 权限、状态和数据条件交织时，应先抽出可读的业务语义变量。
- 禁止将同一权限规则复制到多个模板片段中；需要复用时应抽成统一判断。
- 权限控制不应只依赖前端隐藏，涉及真实安全边界时必须由后端校验。

## 数据映射

- 接口 DTO 与页面模型之间的转换应集中处理，不要散落在模板、表格列和事件回调中。
- 提交前装配、详情回显、枚举展示、金额和日期格式化应有明确位置。
- 页面模型应服务交互表达，不要让接口字段结构直接绑死页面状态结构。

## 状态流转

- 多状态业务不要用多个互斥布尔值拼接状态，优先使用明确的状态字段或状态枚举。
- 存在审批、发布、启停、支付等流程时，应把状态流转条件集中表达。
- 行操作涉及状态机或二次确认时，应抽成明确的业务操作编排，并覆盖异常路径。

# 测试规范

本文件用于约束测试设计和测试质量。具体测试框架、命令和目录位置以项目级规范为准。

## 测试原则

- 新增能力必须补充测试，覆盖正常场景、边界场景和异常场景。
- 修复 bug 时，优先补充能复现问题的回归测试，再实现修复。
- 测试应验证真实业务行为，不应只验证实现细节。
- 不允许为了通过测试而删除、跳过、弱化断言或放宽真实行为。
- 测试数据应表达场景意图，避免使用无意义的 `foo`、`bar`、`test` 作为核心业务数据。

## 覆盖范围

- 输入边界：空值、缺省值、非法值、极值和重复值。
- 状态变化：加载态、成功态、错误态、空态和重置态。
- 异步逻辑：请求成功、请求失败、重复触发、乱序返回和取消场景。
- 权限与条件分支：有权限、无权限、部分权限和配置缺失。
- 用户交互：提交、取消、确认、关闭、切换和连续操作。

## 组件与 Hooks 测试

- 组件测试应覆盖 props、emit、slot、条件渲染和关键交互。
- 支持 `v-model` 的组件应覆盖外部更新、内部更新和循环更新风险。
- hooks 测试应覆盖响应式变化、watch 触发、异步完成、错误回调和清理逻辑。
- 包含计时器、事件监听、订阅或全局副作用的逻辑，必须覆盖释放或卸载后的行为。

## 断言要求

- 断言应面向可观察结果，例如返回值、状态、事件、DOM 输出或副作用结果。
- 不要只断言函数被调用，应同时断言调用参数和结果状态。
- 异步测试必须等待关键状态稳定后再断言。
- 快照测试只能作为辅助，不应替代关键行为断言。

## 测试维护

- 测试命名应描述具体行为，而不是重复实现名称。
- 重构时应保持测试表达的行为不变。
- 当需求变化导致测试需要调整时，应同步更新测试名称、数据和断言，避免测试语义滞后。

# 文档规范

本文件用于约束通用文档写作和维护。具体文档目录、站点构建和发布流程以项目级规范为准。

## 文档原则

- 文档应面向使用者解释能力、边界和正确用法，而不是复述源码实现。
- 公共 API、默认行为、错误行为或使用方式变化时，应同步更新文档。
- 示例应可运行或接近真实使用场景，避免只有伪代码。
- 文档中的术语、参数名、事件名和代码导出应与实现保持一致。
- 文档应优先回答“何时使用、如何使用、边界是什么、出错怎么办”。
- 不要使用营销化、夸张化描述替代具体能力说明。

## README

- README 应说明能力定位、适用场景、基础用法和最小示例。
- README 不应堆叠所有实现细节；复杂 API、设计取舍和边界应拆到独立文档。
- README 中的示例应优先展示推荐用法，而不是边缘能力。

README 至少应覆盖：

- 能力定位：这个模块解决什么问题，不解决什么问题。
- 适用场景：何时应该使用，何时不应该使用。
- 引入方式：如何导入或接入。
- 最小示例：展示最短可理解的推荐用法。
- 常用配置：只列常见配置，不把完整 API 表塞进 README。
- 注意事项：兼容性、依赖、边界行为或易错点。

## API 文档

- API 文档应说明参数、返回值、默认值、错误行为和边界条件。
- 类型复杂时，应解释关键字段语义，而不是只贴类型定义。
- 异步 API 应说明触发时机、完成时机、错误路径和并发行为。
- 组件 API 应说明 props、emits、slots、暴露方法和 `v-model` 更新时机。

每个公开 API 至少应覆盖：

- 签名：函数、组件、hook 或配置项的调用形式。
- 参数：字段语义、是否必填、默认值和约束。
- 返回值：返回结构、状态含义和空值情况。
- 错误行为：抛错、失败返回、忽略输入或降级策略。
- 边界条件：空数据、重复调用、并发、权限不足或非法输入。
- 示例：推荐用法和必要的异常用法。

组件 API 还应说明：

- props 的默认值、受控或非受控行为。
- emits 的触发时机和 payload 结构。
- slots 的用途、作用域参数和空态表现。
- `v-model` 的更新时机、循环更新风险和多 model 场景。
- 暴露方法的调用前置条件和副作用。

## 示例与迁移

- 示例代码应使用清晰命名，避免无意义变量和与真实场景脱节的数据。
- 存在多种用法时，应先展示推荐路径，再说明替代路径。
- 发生破坏性变更或迁移成本较高时，应提供迁移说明。
- 文档不要承诺尚未实现或没有测试覆盖的能力。
- 示例不应依赖未声明的上下文、全局变量或隐藏配置。
- 示例中的类型、字段名、事件名应与当前实现一致。
- 示例应尽量覆盖真实使用链路，而不是只展示孤立片段。
- 迁移说明应包含旧用法、新用法和行为差异。

## 维护要求

- 修改代码时发现文档过期，应在当前影响范围内同步修正。
- 删除能力时，应删除或更新对应文档示例。
- 文档变更后，应按项目级规范运行文档检查或构建命令。

以下变更必须评估是否同步文档：

- 新增、删除或重命名导出。
- 修改函数参数、返回值、类型约束或默认值。
- 修改组件 props、emits、slots、暴露方法或 `v-model` 行为。
- 修改 hook 返回结构、状态含义、异步时序或清理逻辑。
- 修改错误处理、边界输入、权限判断或兼容性要求。
- 修改推荐用法、依赖要求或初始化方式。

## 禁止事项

- 不要写“支持某能力”但代码没有实现或测试没有覆盖。
- 不要保留过期示例、过期参数或已删除 API。
- 不要只写概念说明而缺少最小可用示例。
- 不要把内部实现细节当作使用者必须理解的前置知识。
- 不要用模糊词替代边界说明，例如“自动处理”“智能识别”“完善支持”。
- 不要在文档中引导用户依赖私有 API、内部路径或临时实现。
